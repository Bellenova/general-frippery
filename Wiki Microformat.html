Wireframe Instructions

The wireframe was designed to keep the html as clean and simple as possible. We are moving away from using ids in the html and css because we want to keep the html neutral, in case dev or the client needs to flow different information than what we display in the design phase. The goal is to reduce our workload, as well as enabling us to use work coming directly from the design phase as templates, and to eliminate the huge number of hours used during implementation.

Please follow the wireframe structure as closely as possible. While we are no longer constrained by existing frontend templates, we need to follow the wireframe guidelines closely to produce clean templates and to help dev during implementation.

Navigation

All navigation now uses the same class structure, whether it's top, left, or even the tabs that control widgets. Navigation can be in containers, but please keep the classname generic. In the wireframe, you'll see navigation-1 and navigation-2 containers.

The class structure is as follows: 

<div class="navigation">

        <h3>Topics</h3>
        <ul class="level1">
          <li class="level1-li first"> <a href="#" class="link1">Community</a>

            <div class="sub-menu">
              <ul class="level2">
                <li class="level2-li"><a href="#" class="link1">Navigation Link</a></li>

                <li class="level2-li"><a href="#" class="link2">Navigation Link</a></li>
                <li class="level2-li"><a href="#" class="link3">Navigation Link</a></li>

                <li class="level2-li"><a href="#" class="link4">Navigation Link</a></li>
                <li class="level2-li"><a href="#" class="link5">Navigation Link</a></li>

                <li class="level2-li"><a href="#" class="link6">Navigation Link</a></li>
                <li class="level2-li"><a href="#" class="link7">Navigation Link</a></li>

              </ul>
            </div>
          </li>
          <li class="level1-li last"> <a href="#" class="link2">Store Locator</a> </li>

        </ul>
      </div>

This differs from our current structure in that class names are assigned to every element in the navigation structure. This makes it much easier to style in the css, and to modify using javascript. Links have incremented classes for that same reason. If you wish to style or control an entire block of links, you can call the element. Additionally, you can display multiple columns in the submenu, which is done entirely through css (see wireframe for an example.)

Also, you will see the addition of a div around the submenu. This makes styling the submenu much easier.

Layout

The layout is often divided into two sections - the body and everything else. The body, according to our traditional design standards, is often centered in the middle of the browser. Everything else includes the logo, the top navigation(s), any masthead, footer, or any other element that can't be considered part of the main body column. These elements are contained in descriptive classes (logo, masthead, footer, navigation-1). All of these elements should be free floating, unless they are to be centered with the body.

Elements that appear in the body column should be inside the .document class. This class is used primarily for positioning. There is an additional class called .container, that should have any styles associated with the main content area applied. In the wireframe, quick-search appears between the .document class and .container class - this is fine, since we still want quick search to be part of the body column but not inside the .container class.

Sections

Inside .container, you will see sequentially named classes called #section-1, #section-2, etc. These are our old "columns". Since we don't always design in the standard three column format, these layers are designed to exist in any shape or location, yet allow us to easily break up the page. They tend to run from right to left, top to bottom, exactly like the old "column" styles. Any number of sections may be added or subtracted to your comps, but please make sure they are sequential to allow someone who's unfamiliar with your design to easily modify the javascript/css.

Boxes

Each content block within a section has a sequentially named class called .box1, .box2, etc. for the same reason listed above. Boxes can have any kind of content, from navigation to polls, to a record. This comes in handy for right columns, where individual sections need special treatment, but often have the same layout.

In the example above, you can see that additional classes have been appended to the generic record structure. This is solely to conform with the hcalendar standard. All events should have this structure, but extra classes should not be appended to any other format.

Each .records class can contain several .record classes. It should not contain anything else. The .record class can contain absolutely anything.

Type Classes

You can further refine your css if you wish by adding type classes to the .records div: 


<div class="records event">
<div class="record">...</div>
</div>

This allows us the flexibility to control all events, articles, links, or other item that have this marker.

JQuery

There is a library called Jquery Tools installed by default. This provides a wide range of effects in one script. Less includes means less load time.

http://flowplayer.org/tools/index.html

Before adding a jquery plugin, check to see if the effect can be done with this library. If there are any questions about what can or cannot be done with this library, please feel free to ask.
